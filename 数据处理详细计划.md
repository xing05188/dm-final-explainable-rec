## 1. 字段提取

### 1.1 Amazon Review 2018（Electronics_5.json.gz）全部字段

每条 review 是一行 JSON，**2018 版**官方字段如下（与 2014 版的 `helpful` 不同，2018 版用 `vote`）：

| 字段 | 类型 | 示例 | 说明 |
| --- | --- | --- | --- |
| `reviewerID` | string | `"A2SUAM1J3GNN3B"` | 用户唯一 ID |
| `asin` | string | `"B0000012345"` | 商品唯一 ID（Amazon Standard Identification Number） |
| `reviewerName` | string | `"J. McDonald"` | 用户昵称（含隐私信息） |
| `overall` | float | `5.0` | 评分，1–5 星 |
| `reviewText` | string | `"Great product, works well..."` | 评论正文 |
| `summary` | string | `"Excellent battery life"` | 评论标题/摘要 |
| `unixReviewTime` | int | `1252800000` | Unix 时间戳（秒） |
| `reviewTime` | string | `"09 13, 2009"` | 可读日期（与 unixReviewTime 冗余） |
| `verified` | bool | `true` | 是否为 Verified Purchase |
| `vote` | string/int | `"2"` 或 `5` | 该评论被标记"有用"的次数 |
| `style` | dict | `{"Color:": "Black", "Size:": "Large"}` | 用户购买的具体规格变体 |
| `image` | list[string] | `["https://images-na.ssl-images-amazon.com/..."]` | 用户上传的晒图 URL |

> 另有独立的 **meta 文件**（`meta_Electronics.json.gz`），含 `title`、`brand`、`categories`、`description` 等商品元数据。本项目**不使用 meta 文件**，仅依赖 review 文件中的交互 + 文本。

### 1.2 字段选用决策（团队确认）

| 字段 | 是否使用 | 映射后列名 | 用途 | 理由 |
| --- | --- | --- | --- | --- |
| `reviewerID` | **是** | `user_id` | 用户标识、协同过滤、图构建 | 核心主键 |
| `asin` | **是** | `item_id` | 商品标识 | 核心主键 |
| `overall` | **是** | `rating` | CF/NCF 评分预测、隐式反馈 | 核心交互信号 |
| `reviewText` | **是** | `review_text` | SBERT 编码、关键词解释 | RQ2 语义增强 + 可解释性 |
| `summary` | **是** | `summary` | 与 reviewText 拼接后送 SBERT | 短文本补充语义，几乎每条都有 |
| `unixReviewTime` | **是** | `timestamp` | Leave-One-Out 时序划分 | 比 reviewTime 更规范 |
| `reviewerName` | 否 | — | — | 隐私敏感，对推荐无增益 |
| `reviewTime` | 否 | — | — | 与 unixReviewTime 冗余 |
| `verified` | 否 | — | — | 可选质量过滤，增加复杂度，K-core 已足够 |
| `vote` | 否 | — | — | 评论质量信号，本项目不建模 |
| `style` | 否 | — | — | 商品变体信息，Electronics 稀疏且非必需 |
| `image` | 否 | — | — | 多模态超出项目范围 |

**最终保留 6 列**：`user_id, item_id, rating, review_text, summary, timestamp`

对应 `config.py` 中的字段配置：

```python
"fields": {
  "user_id": "reviewerID",
  "item_id": "asin",
  "rating": "overall",
  "review_text": "reviewText",
  "summary": "summary",
  "timestamp": "unixReviewTime",
}
```

### 1.3 清洗规则（与字段决策一致）

```plain
- 删除 reviewText 为空或仅空白的行（summary 可为空，Stage 7 会拼接处理）
- 删除 overall 不在 1–5 范围的行
- 删除 unixReviewTime 缺失或为 0 的行
- timestamp 转为 int
```

## 2. 数据清洗
### Step 1：基础清洗
```plain
- 删除 review_text 为空的行
- 删除 rating 不在 1-5 范围的行
- 删除 timestamp 为空的行
- timestamp 转为 int
```

### Step 2：K-core 过滤
```plain
迭代过滤（通常 2-3 轮收敛）:

while True:
    过滤: 保留交互数 >= 5 的用户
    过滤: 保留交互数 >= 5 的商品
    if 双方数量都不再变化: break
```

### Step 3：用户/商品 ID 重编码
做完过滤后，原始 ID（字符串）映射为连续的 0 ~ N-1：

```python
user_ids  = {原始ID: 0, 1, 2, ..., n_users-1}
item_ids  = {原始ID: 0, 1, 2, ..., n_items-1}
```

## 3. 采样策略

K-core 过滤后通常还有数十万用户，不需要全用。当前实现见 `config.py` 中 `filter.sample_active_users` 等参数。

### 3.1 当前策略（已实现）

```python
# 1. 选取交互数最多的 Top-5000 用户（非随机）
# 2. 保留这些用户交互过的所有商品
# 3. 只保留被 >= 2 个不同用户交互过的商品（去孤立商品）
# 4. 若商品数 > 3000：按交互次数保留 Top-3000 商品
# 5. 剔除交互数 < 5 的用户；再次去掉孤立商品（迭代至多 5 轮）
```

**产出（全量数据实测）**：~5000 用户 × 3000 商品 × **~11 万条交互**（人均约 20+ 条）

对应配置：

```python
"filter": {
    "n_users_sample": 5000,
    "n_items_target": 3000,
    "min_interactions_per_user": 5,   # 采样后质量过滤
    "sample_active_users": True,      # True=Top活跃用户；False=随机用户
}
```

### 3.2 采样策略更换说明

| 项目 | 初版（已废弃） | 当前版（在用） |
| --- | --- | --- |
| 用户抽样 | `np.random.choice` 随机 5000 用户 | `user_activity.nlargest(5000)` **交互最多的 Top-5000 用户** |
| 商品裁剪 | Top-3000 热门商品 | 相同 |
| 用户质量过滤 | 无 | 商品裁剪后剔除交互 **< 5** 的用户 |
| 全量 Pipeline 实测交互数 | **17,922** | **116,256** |
| 实测用户数 | 4,634 | 4,964 |
| 人均交互 | **3.9** | **23.4** |

**更换时间**：全量 `Electronics_5.json.gz`（约 674 万条）首次跑通 Pipeline 后，发现初版采样过稀，于 **Stage 3 修正代码并重跑 Stage 3–8**（Stage 1–2 缓存保留）。

**更换原因**：

1. 随机用户的历史商品分散，再只保留子图内 Top-3000 热门商品时，大量交互被裁掉，子图极稀疏。  
2. NCF / LightGCN / 协同过滤需要足够密的 user-item 边，人均不足 5 条时难以训练、指标不稳定。  
3. Top 活跃用户子集是推荐实验常见做法，仍满足「约 5000×3000、5–10 万+ 交互」的项目目标。

**结论**：文档与代码均以 **§3.1 当前策略** 为准；勿再使用随机用户初版。若需复现初版行为，可在 `config.py` 设 `sample_active_users: False`（不推荐）。

## 4. 数据划分（Leave-One-Out by Time）
推荐系统**不能随机 shuffle 划分**，必须按时间顺序：

```python
def split_by_time(df):
    """每个用户最后1条→test, 倒数第2条→val, 其余→train"""
    train_list, val_list, test_list = [], [], []

    for user_id, group in df.groupby("user_id"):
        group = group.sort_values("timestamp")
        # 至少要有 3 条交互才够分
        if len(group) >= 3:
            train_list.append(group.iloc[:-2])
            val_list.append(group.iloc[-2:-1])
            test_list.append(group.iloc[-1:])

    return pd.concat(train_list), pd.concat(val_list), pd.concat(test_list)
```

划分后：

```plain
Train set:    ~80-85%  用于训练模型参数
Val set:      ~7-10%   用于调超参数 / Early Stop
Test set:     ~7-10%   用于最终报告指标（只跑一次）
```

## 5. 负采样
NCF 和 LightGCN 训的是二分类（正=有交互，负=无交互），需要生成负样本。

### 正样本
train 中**每一行交互** → label = 1（同一用户对同一商品多条评论会占多行）

### 负样本
```python
def sample_negatives(train_df, n_items, neg_ratio=4):
    """每 1 行正样本配 neg_ratio 个负样本（按交互行计，非按 unique item）"""
    for user_id, group in train_df.groupby("user_id"):
        pos_set = set(group["item_id"])
        neg_candidates = list(all_items - pos_set)
        n_neg = len(group) * neg_ratio   # len(group)=该用户 train 行数
        ...
```

> **说明**：若按 `unique(item)` 计负样本，当同一 (user, item) 有多条评论时，正负比会低于 1:4。Pipeline 按**交互行**计数生成 `train_neg.csv`（与正样本严格 1:4，供检查与 `resample_per_epoch=False` 时使用）。

**关键**：每个 epoch 重新采样负样本，保证模型见到足够多样的负例。

| 模型 | 实现位置 | 行为 |
| --- | --- | --- |
| **NCF / NCF+Review** | `src/ncf.py`、`src/ncf_review.py` | `resample_per_epoch=True` 时，每 epoch 调用 `sample_train_negatives()`（`seed = config.seed + epoch`） |
| **LightGCN** | `src/lightgcn.py` `BPRDataset.__getitem__` | 每个正样本在取 batch 时随机抽负样本，等价于每 epoch 重采 |
| **Pipeline Stage 5** | `src/preprocess.py` | 写出一份 `train_neg.csv` 快照，不替代训练期重采 |

训练入口传入 **`train.csv`（仅正样本）** 即可；不必在训练脚本里读 `train_neg.csv`，除非设 `resample_per_epoch: False`。

## 6. 评论文本预处理（D 负责）

清洗目标是**爬虫与平台模板带来的噪音**，不删 "the / is" 等通用停用词（SBERT 不敏感；关键词解释在 `explain.py` 单独去停用词）。

```python
def clean_review_text(text):
    # 1. HTML 实体与标签: <br>, <div>, &amp; ...
    # 2. URL
    # 3. 固定前缀: "verified purchase", "amazon vine ..." 等（见 config.text.strip_prefixes）
    # 4. 连续重复字符 (aaaaa) 与长串符号乱码
    # 5. 小写 + 仅保留字母数字，合并空白
```

实现：`src/utils.py` → `clean_review_text()`；Stage 7 对 `review_text` 与 `summary` 均调用。

### Step 2：停用词过滤（不做）

不在 Pipeline 中删停用词。可选逻辑仅用于 `KeywordExplainer` 展示层。

### Step 3：截断（头尾各半）

聚合后总长度上限 `max_tokens × 4` 词（默认 512）。`truncate_mode: head_tail`：**前半保留最早语境，后半保留最近结论**（非只留头部）。

单条评论过长时先按 `max_words_per_review`（默认 96）做头尾截断，再参与合并。

### Step 4：聚合（按时间智能合并）

```python
# 1. 组内按 timestamp 升序（先旧后新）
# 2. 每条: summary 作标题 + review_text，若 summary 已在正文中则去重
# 3. 完全相同片段去重
# 4. 用 " . " 连接各条评论
# 5. 对合并结果再做 head_tail 截断 → user_reviews.json / item_reviews.json
```

实现：`src/utils.py` → `compose_review_snippet()`、`aggregate_review_texts()`。

### Step 5：SBERT 编码

from sentence_transformers import SentenceTransformer

## 8. 数据处理 Pipeline（必须）
**不允许手动逐步跑脚本。** 必须封装成一个端到端 Pipeline，一次命令从头跑到尾。

### 8.1 Pipeline 设计原则
```plain
一键复现：任何人拿到代码和数据，一行命令出结果
断点续跑：中间步骤的结果缓存到磁盘，重跑时跳过已完成的步骤
每步校验：每个阶段跑完自动打印数据形状和统计量，异常立刻报错
配置分离：参数写在 config 里，不硬编码在脚本中
```

### 8.2 Pipeline 阶段
```plain
Stage 1: 原始数据加载
  input:  data/raw/Electronics_5.json.gz
  output: data/processed/raw_interactions.csv
  校验:  打印行数、列名、rating 分布

Stage 2: 数据清洗 + K-core 过滤
  input:  data/processed/raw_interactions.csv
  output: data/processed/cleaned_interactions.csv
  校验:  打印 n_users, n_items, 每轮过滤变化量

Stage 3: ID 重编码 + 采样
  input:  data/processed/cleaned_interactions.csv
  output: data/processed/interactions.csv
          data/processed/user_map.json
          data/processed/item_map.json
  校验:  打印最终 n_users, n_items, n_interactions, sparsity

Stage 4: Leave-One-Out 划分
  input:  data/processed/interactions.csv
  output: data/processed/train.csv
          data/processed/val.csv
          data/processed/test.csv
  校验:  打印每个集合行数、val 和 test 的 user 是否都在 train 中

Stage 5: 负采样
  input:  data/processed/train.csv
  output: data/processed/train_neg.csv
  校验:  打印正负样本数、正负比

Stage 6: 统计信息
  input:  data/processed/interactions.csv + train.csv + val.csv + test.csv
  output: data/processed/stats.json
  校验:  打印 stats.json 内容

Stage 7: 评论文本预处理（D 负责）
  input:  data/processed/interactions.csv (review_text 列)
  output: data/processed/user_reviews.json   # {user_id: "聚合文本"}
          data/processed/item_reviews.json   # {item_id: "聚合文本"}

Stage 8: SBERT 编码（D 负责）
  input:  data/processed/user_reviews.json + item_reviews.json
  output: data/processed/user_emb.npy
          data/processed/item_emb.npy
  校验:  打印 embedding shape，确认 user_emb[i] 和 item_emb[j] 无 NaN
```

### 8.4 入口脚本 `src/preprocess.py`
```python
"""一键跑通完整数据处理 Pipeline"""
import config

def run_pipeline():
    stages = [
        ("1. 加载原始数据", load_raw_data),
        ("2. 清洗 + K-core 过滤", clean_and_filter),
        ("3. ID重编码 + 采样", reindex_and_sample),
        ("4. Leave-One-Out 划分", split_train_val_test),
        ("5. 负采样", negative_sampling),
        ("6. 统计信息", compute_stats),
        ("7. 文本预处理", preprocess_text),
        ("8. SBERT 编码", encode_sbert),
    ]

    for name, stage_fn in stages:
        print(f"\n{'='*50}")
        print(f"  {name}")
        print(f"{'='*50}")
        stage_fn()
        print(f"  ✓ {name} 完成")

    print("\n Pipeline 全部完成! 产出在 data/processed/")

if __name__ == "__main__":
    run_pipeline()
```

### 8.5 断点续跑
每个 Stage 写完产出文件后，在 `data/processed/.pipeline_state` 记录已完成的 Stage。下次运行自动跳过：

```python
def should_skip(stage_name):
    return stage_name in completed_stages

# 每个 stage 开头:
if should_skip("Stage 3"):
    print("  → 已缓存，跳过")
    return
```

### 8.6 为什么必须做成 Pipeline
| 没有 Pipeline | 有 Pipeline |
| --- | --- |
| 每人手动跑，顺序可能搞错 | 一行命令，顺序固定 |
| 参数改了一个脚本忘记改另一个 | 所有参数在 config.py，改一次生效全局 |
| 某步报错不知道哪出的问题 | 每步校验，立刻定位 |
| 重跑浪费时间 | 断点续跑，只重跑改动的部分 |
| 换台电脑/换个人跑不通 | 依赖 config + 校验，环境问题立刻暴露 |
| 实验不可复现 | 固定 seed + 缓存中间结果 → 完全可复现 |
