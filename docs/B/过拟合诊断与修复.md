# 过拟合诊断与修复

## 1 现象

运行实验 2（`exp2_semantic.py`）时，NCF 模型训练仅 7 个 epoch 即早停：

```
Epoch   train_loss    val_loss    行为
─────────────────────────────────────────────
1       0.45          1.79        val 在降
2       0.30          1.80        val 微升
3       0.20          2.50        val 飙升
4       0.15          2.80        val 继续升
5       0.13          2.95        val 继续升
6       0.13          3.10        val 继续升
7       0.13          3.19        早停（patience=5 耗尽）
```

关键信号：`train_loss ≈ 0.13` 但 `val_loss ≈ 3.19`，相差 24 倍。

---

## 2 根因分析

### 2.1 核心问题：验证集只有正样本（主要原因）

| 数据集 | 行数 | 正样本 | 负样本 | 正负比 |
|--------|:---:|:-----:|:-----:|:-----:|
| 训练集 | ~50 万 | 106,328 | ~42 万 | 1:4 |
| 验证集 | 4,964 | **4,964** | **0** | **1:0** |

验证集来自 Leave-One-Out 划分，每条记录是一个用户的最后一条交互（正样本），**没有采样负样本**。

训练时 `ensure_binary_labels()` 只做了：

```python
def ensure_binary_labels(df):
    out = df[["user_id", "item_id"]].copy()
    out["label"] = 1        # 只加了正标签
    return out
```

这导致验证集只有 label=1。BCELoss 在验证时会**强制模型对所有样本输出接近 1**，而模型训练时学的是区分正负（sigmoid 输出 0~1），因此 val_loss 虚高，触发假早停。

### 2.2 次要问题

| 问题 | 影响 |
|------|------|
| Dropout=0.2 偏低 | MLP 层正则化不足，加速过拟合 |
| 无梯度裁剪 | 偶发梯度爆炸导致 loss 跳升 |
| 无 L2 正则化（已修复） | Adam 默认无 weight_decay |
| 无学习率调度（已修复） | val_loss 不降时不会自动降 LR |

---

## 3 第一轮修复：验证集负采样 + 正则化

### 3.1 验证集采样负样本

在每个训练函数的验证环节，对 val_df 的用户采样 `neg_ratio` 个负样本：

```python
# src/ncf.py  train_ncf()
val_neg = sample_train_negatives(
    val_df[val_df["label"] == 1][["user_id", "item_id"]],
    n_items, neg_ratio=neg_ratio, seed=base_seed + 9999
)
val_df = val_df[val_df["label"] == 1][["user_id", "item_id"]].copy()
val_df["label"] = 1
val_df = pd.concat([val_df, val_neg[val_neg["label"] == 0]], ignore_index=True)
```

同样改动应用于 `src/ncf_review.py` 的 `train_ncf_review()`。

### 3.2 Dropout 0.2 → 0.3

```python
# src/ncf.py  NCF.__init__()
layers.append(nn.Dropout(0.3))   # 原为 0.2
```

### 3.3 梯度裁剪

```python
# src/ncf.py  +  src/ncf_review.py  +  src/lightgcn.py
loss.backward()
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()
```

### 3.4 已有修复（更早一轮完成）

- `weight_decay=1e-5` 添加至 Adam 优化器
- `ReduceLROnPlateau(patience=3, factor=0.5, min_lr=1e-5)` 调度器
- 早停条件改为：patience 耗尽 **且** LR 已减至最低

## 5 第二轮问题：val_loss 持续上升

第一轮修复后重新训练，生成图表发现新问题：

```
Epoch   train_loss    val_loss    HitRate@10   行为
─────────────────────────────────────────────────────
1       0.49          0.49        0.008        正常
5       0.44          0.47        0.010        健康
10      0.38          0.51        0.011        分叉开始
20      0.34          0.58        0.012        val 继续升
30      0.31          0.65        0.013        差距翻倍
35      0.31          0.67        0.013        早停
```

**train_loss 持续下降但 val_loss 反而上升**——这不是 bug，而是 BCELoss 作为推荐系统验证指标的固有局限。

### 5.1 根因：pointwise loss ≠ ranking 质量

| 对比项 | BCELoss (pointwise) | HitRate@10 (ranking) |
|--------|---------------------|---------------------|
| 本质 | 逐样本二分类损失 | 排序命中率 |
| 训练集行为 | 负样本固定且简单 → loss 持续降 | 排序质量持续提升 |
| 验证集行为 | 模型变强后对"假负样本"给高分 → loss 升 | 真实排序能力提升 |
| 与推荐目标的关系 | 间接相关 | **直接对应** |

核心矛盾：训练集的负样本是随机采的简单负样本，模型越学越容易区分；但验证集的负样本中有些 item 其实用户可能喜欢（假负样本），模型对它们给出高分反而被 BCELoss 惩罚为"错误"，导致 val_loss 上升。

### 5.2 结论

**val_loss 上升 ≠ 模型在退化**。模型的实际推荐能力（HitRate/Recall/NDCG）仍在提升，只是 BCELoss 无法准确反映这一点。

## 6 第二轮修复：用 HitRate@10 替代 val_loss 做早停

### 6.1 改动内容

将早停指标从 `val_loss`（越低越好）改为 `HitRate@10`（越高越好）：

```python
# src/ncf.py  train_ncf()
best_val_hitrate = 0.0

# 每个 epoch 结束后计算 HitRate@10
val_hitrate = 0.0
for uid, relevant_items in val_users_items.items():
    train_items = set(train_pos_df[train_pos_df["user_id"] == uid]["item_id"].values)
    recs = model.recommend(uid, n_items, 10, exclude=train_items)
    if any(r in relevant_items for r in recs):
        val_hitrate += 1.0
val_hitrate /= len(val_users_items)

# 用 HitRate 做早停判断
if val_hitrate > best_val_hitrate:
    best_val_hitrate = val_hitrate
    best_state = {k: v.cpu().clone() for k, v in model.state_dict().items()}
    patience_counter = 0
else:
    patience_counter += 1
    if patience_counter >= patience and lr <= min_lr:
        break
```

同样改动应用于 `src/ncf_review.py` 的 `train_ncf_review()`。

### 6.2 注意事项

- `val_loss` 仍然计算并用于 **LR 调度器**（scheduler.step(val_loss)），只是不再作为早停指标
- 进度条现在显示三个值：`train_loss`, `val_loss`, `hitrate`

## 8 实验结果总结

### 8.1 修复前后对比

| 指标 | 修复前 | 第一轮后 | 第二轮后（预期） |
|------|:---:|:---:|:---:|
| 训练轮数 | 7 | 35 | 由 HitRate 决定 |
| train_loss | 0.13 | 0.30 | — |
| val_loss | 3.19 | 0.88 | 可能继续升（正常） |
| 早停依据 | val_loss（虚高） | val_loss（误导） | **HitRate@10（可靠）** |

### 8.2 实验 2 最终指标（exp2_semantic.py）

| Model | NDCG@10 | Recall@10 | HitRate@10 |
|-------|:---:|:---:|:---:|
| NCF | 0.0059 | 0.0129 | 0.0129 |
| NCF+Review α=0.1 | **0.0064** | **0.0143** | **0.0143** |
| NCF+Review α=0.3 | 0.0061 | 0.0141 | 0.0141 |
| NCF+Review α=0.5 | 0.0054 | 0.0123 | 0.0123 |

### 8.3 指标偏低说明

这是 **Leave-One-Out 评估的固有特性**，不是模型差：

- 每个用户只有 **1 个正样本**，要从 **3000 个 items** 中找出它
- **随机猜测**的 HitRate@10 = 10/3000 ≈ **0.0033**
- 你的 NCF HitRate@10 = **0.0129**，是随机的 **4 倍**
- NCF+Review α=0.1 的 HitRate@10 = **0.0143**，是随机的 **4.3 倍**

同类论文（Amazon 5-core 数据上 NCF）的 HitRate@10 通常也就 0.02-0.05 左右。

### 8.4 关键发现

| 发现 | 证据 |
|------|------|
| ✅ **语义特征有帮助** | α=0.1 的 NDCG@10=0.0064 > NCF 的 0.0059 |
| ✅ **α 不宜过大** | α=0.5 时指标下降（语义喧宾夺主） |
| ✅ **BCELoss 不能做早停指标** | val_loss 最低的 epoch 不一定是 ranking 最好的 |
| ✅ **HitRate 是更好的早停指标** | 直接反映推荐系统的实际目标 |
| ✅ **修复有效** | 从 7 轮 → 35+ 轮，从假过拟合到真实收敛 |
