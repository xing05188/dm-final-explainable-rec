# 1. 项目背景（Project Background）
随着电子商务与在线内容平台的快速发展，推荐系统已经成为互联网核心技术之一。

传统推荐系统（如协同过滤）虽然能够基于用户历史行为进行推荐，但存在以下问题：

+ 无法理解商品语义
+ 无法解释推荐原因
+ 难以建模复杂用户关系
+ 冷启动问题严重

与此同时，大量用户评论文本中蕴含着丰富的用户偏好信息，例如：

+ 用户关注商品哪些特征
+ 用户真正喜欢/不喜欢什么
+ 商品潜在语义属性

因此，本项目尝试融合：

+ 用户行为数据
+ 评论文本语义
+ 图结构关系

构建一个：

> 可解释（Explainable）、语义增强（Semantic-aware）、图建模（Graph-based）的智能推荐系统。
>

# 5. 数据集设计（Dataset）
## 主数据集
Amazon Review Dataset — **Electronics 子集**

### 为什么选 Electronics
+ 社区使用最多，预处理代码随处可参考
+ 评论文本质量高（用户会明确写"电池好""降噪强"，天然适合关键词解释）
+ 下载即用，无需爬虫或 API Key

### 数据字段
| 字段 | 用途 |
| --- | --- |
| user_id | 用户建模 |
| item_id | 商品建模 |
| rating | 用户偏好 |
| review_text | NLP分析 |
| timestamp | 时序分析 |
| category | 商品属性 |
| title | 商品语义 |


---

## 5.1 数据采样策略（关键）
全量 Electronics ~170 万条交互，没必要。本项目采用 K-core 过滤 + 随机采样：

```latex
1. 过滤交互数 >= 5 的用户
2. 过滤交互数 >= 5 的商品
3. 随机采样 5000 用户 × 3000 商品
4. 保留这些用户的所有交互（约 5-10 万条）
```

=> 最终数据量：~5-10 万条交互，笔记本 CPU 完全够用。

---

## 5.2 数据划分策略（重要）
推荐系统不能随机划分！必须按时序划分，否则数据泄露导致评估虚高：

```latex
对于每个用户：
  - 最后 1 条交互 → Test Set
  - 倒数第 2 条交互 → Validation Set
  - 其余 → Train Set
```

这种 Leave-One-Out 方式是 RecSys 领域标准做法。

---

## 5.3 负采样策略
NCF 和 LightGCN 训练需要正负样本：

```latex
正样本：用户实际交互过的商品
负样本：随机采样用户未交互过的商品（每正样本采样 4 个负样本）
```

每轮 epoch 重新随机采样负样本，保证模型见到足够多样的负例。

---

## 5.4 评论文本预处理
```latex
原始评论文本
  → 转小写
  → 去除特殊字符和标点
  → 去除停用词
  → 截断到 128 token（Sentence-BERT 足够）
  → 聚合：同一用户的所有评论拼接 → 用户偏好文本
           同一商品的所有评论拼接 → 商品特征文本
```

聚合后的文本输入 Sentence-BERT 得到 embedding。

## 7.3 创新模型
### NCF + Review Embedding（融合方式详解）
两种 embedding 的融合方式：

```latex
Step 1: NCF 生成行为嵌入
  user_emb (dim=64) + item_emb (dim=64) → MLP → behavior_vec (dim=64)

Step 2: Sentence-BERT 生成语义嵌入
  user_review_emb (dim=384) + item_review_emb (dim=384)
  → 线性投影到 64 维 → semantic_vec (dim=64)

Step 3: 加权融合
  final_vec = (1 - α) × behavior_vec + α × semantic_vec
  其中 α 是融合权重，默认 0.3，可由实验调优

Step 4: 预测层
  final_vec → Linear(64→32) → ReLU → Linear(32→1) → Sigmoid → 偏好分数
```

### LightGCN
**原理：** 在 User-Item 二部图上做 embedding 传播，每层聚合邻居信息。

```latex
输入：User-Item 交互图（稀疏邻接矩阵）
过程：逐层传播
  Layer 0: E(0) = 初始 embedding（随机初始化，dim=64）
  Layer 1: E(1) = 聚合 1 阶邻居
  Layer 2: E(2) = 聚合 2 阶邻居
  Layer 3: E(3) = 聚合 3 阶邻居
输出：各层 embedding 的加权平均 = 最终 embedding

最终预测分数 = user_final · item_final（内积）
```

**实现：** 纯 PyTorch + 稀疏矩阵乘法，无需 PyTorch Geometric。核心代码 ~50 行。

# 10. Explainability 模块设计
## 10.1 关键词解释（实现最简单）
**原理：** 提取用户历史评论和候选商品评论的高频关键词，计算重叠度。

```python
# 伪代码
user_keywords = tfidf_extract(user.reviews, top_k=10)
item_keywords = tfidf_extract(item.reviews, top_k=10)
overlap = user_keywords & item_keywords  # 交集

explanation = f"推荐原因：你关注的「{overlap[0]}」「{overlap[1]}」「{overlap[2]}」与该商品评论高度重合"
```

纯文本处理，无需模型。TF-IDF + jieba/nltk 即搞定。

## 10.2 相似用户解释（零额外成本）
**原理：** 复用 UserCF 的最近邻结果。

```latex
推荐 SONY WH-1000XM5
原因：与您偏好最相似的 3 位用户中，有 2 位给该商品打了 5 星。
```

UserCF 训练时就计算好了用户相似度，直接查表即可，无额外计算。

## 10.3 SHAP 特征贡献解释（最有学术分量）
**原理：** 用 SHAP 分析每个特征对推荐分数的贡献方向与大小。

```python
import shap

# 构建解释器（用训练集采 500 条做背景）
explainer = shap.KernelExplainer(model.predict, background_samples)

# 解释某条推荐
shap_values = explainer.shap_values(features_for_user_item_pair)

# 自带可视化
shap.waterfall_plot(shap_values[0])  # 单条推荐的推高/拉低分解
shap.summary_plot(shap_values_all)   # 全局特征重要性排名
```

SHAP 的 Waterfall Plot 和 Summary Plot 视觉效果专业，直接放 PPT 很加分。

## 10.4 三种解释方法的定位
| 方法 | 难度 | 学术分量 | 用户体验 | 适用场景 |
| --- | --- | --- | --- | --- |
| 关键词解释 | 低 | 中 | 直观易懂 | 所有推荐 |
| 相似用户解释 | 低 | 低 | 有说服力 | 协同过滤结果 |
| SHAP | 中 | 高 | 需要理解 | 深度模型结果 |


**建议：** 三种都做，按推荐结果类型选择展示哪种解释。关键词 + 相似用户作为默认展示，SHAP 作为"深度分析"功能。

# 15. 风险与应对（Risk & Mitigation）
| 风险 | 概率 | 影响 | 应对方案 |
| --- | --- | --- | --- |
| LightGCN 收敛不好 | 中 | 中 | 降级为 MF + Review Emb 对比，同样是两个创新模型 |
| 数据预处理耗时过长 | 低 | 中 | 提前写采样脚本，先跑通再调参 |
| SHAP 在小模型上效果不明显 | 中 | 低 | 关键词 + 相似用户解释兜底，SHAP 不行也有两种解释可用 |
| 时间不够 | 低 | 高 | **第 6 周是缓冲周**，Phase 4 完成即可交付，Phase 5 是加分 |
| 某个模型调不出来 | 低 | 低 | 模型线是递进的，Baseline 一定跑得通，往上加就是 |


**核心原则：** 每个阶段产出都可以独立展示。3 个 Baseline 跑通了就是一个完整的推荐系统，后面的都是加分项。
